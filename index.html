Serverless? Oy.

Let's get this out of the way: obviously, there are servers involved.

Serverless? Ok, then what?

  • Deploy code in stateless containers
  • Instantly spin up new containers as needed, and destroy when not = inherent horizontal scalability, no idle instances, no idle costs.
  • Heavily async, and event-driven.
  • Like Heroku/PaaS, but not just for http events.
  • AWS Lambdas (Python, Node.js, Java), Google Cloud Functions (Node.js), and Azure Functions (JavaScript, C#, Python, PHP, Bash, .BAT, and PowerShell, among others).

Yeah, but... WHY?

  • Serverless is great for:
    • CPU and I/O heavy-lifting - image and other media processing
    • absorbing huge traffic bursts
    • moving tasks out of your main request loop and into a managed queue
    • small personal projects that are idle most of the time

AWS Lambdas

  • A function (Python, JS, Java) which receives event and context parameters, triggered by an AWS event:
    • S3, DynamoDB, API Gateway, Email, Kinesis, CodeCommit, even Echo/Alexa.
    • Think of it like SQL stored procedures + triggers, except instead of a database it's your AWS infrastructure.
  • ~100ms spinup time; even shorter when kept warm.
  • Functions are called immediately in parallel, or on a schedule (like cron).
  • You pay for each function call, as well as function runtime (in 100ms increments). 1M calls and 400,000 GB-seconds free each month.

Zappa.io - making Lambdas awesome.

  • CLI Python tools for packaging and deploying/managing Lambdas
    • Python dependency packaging
    • CloudFormation, IAM policies
    • Route53 and LetsEncrypt
    • Logging, monitoring, cleanup/undeploy
    • Multiple stages, eg. dev/prod
  • Also adapts any WSGI stack, like Flask or Django, on API Gateway + Lambda

  • Creates a simple Web interface to the watermarked images.
